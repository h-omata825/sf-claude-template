## 1. 全体フロー

### 新規案件

```
プロジェクト作成 → 組織認証 → CLAUDE.md記入 → /sf-analyze
  → 資料読み込み → MCP設定 → Git初回push → メンバー招待
```

### 保守案件

```
プロジェクト作成 → 組織認証 → メタデータ取得 → /sf-analyze
  → 既存資料読み込み → CLAUDE.md記入 → MCP設定 → Git初回push → メンバー招待
```

保守案件は「既に動いている組織」があるため、まず組織の現状把握を先に行う。

## 2. 共通手順: プロジェクト作成

### Step 2-1. プロジェクトフォルダの作成

Claude Codeで以下を実行:

```
/setup-sf-project
```

対話に従って入力:
1. **フォルダ名**: プロジェクト識別名（英語。例: `CaseManagement`, `SalesPortal`）
2. **作成先パス**: ローカルの作業フォルダ（例: `C:/workspace`）
3. **テンプレートURL**: デフォルトでOK
4. **組織認証**: 開発Sandbox or 本番 or スキップ

### Step 2-2. VSCodeで開く

セットアップコマンド完了後、自動でVSCodeが開く。
開かない場合は手動で:

```bash
code <作成先パス>/<フォルダ名>
```

> **重要**: 必ずプロジェクトのルートフォルダを開く（`CLAUDE.md` がある階層）。

### Step 2-3. MCP設定（必要な場合）

```
/setup-mcp
```

GitHub / Slack / Notion 等、プロジェクトで使うMCPのトークンを対話的に設定する。

## 3. パターンA: 新規案件

まだSalesforce組織が空 or ほぼ未設定の状態から始める案件。

### Step A-1. CLAUDE.md にプロジェクト基本情報を記入

`CLAUDE.md`（ルート）を開いて、わかっている範囲で記入する:

```markdown
## Salesforce組織情報
| 環境 | org alias | 用途 |
|---|---|---|
| 開発 | `myproject-dev` | 開発・動作確認 |

## 命名規則
| 対象 | ルール | 例 |
|---|---|---|
| カスタムオブジェクト | `MP_` プレフィックス | `MP_Order__c` |
| Apexクラス | `MP` プレフィックス | `MPOrderService` |
```

> この時点では全部埋まらなくてOK。わかる範囲だけ記入し、プロジェクト進行に伴い追記する。

### Step A-2. 既存資料の読み込み

企画書・提案書・ヒアリングメモ等、プロジェクトの背景資料があれば読み込ませる:

```
/save-doc
（ファイルを貼り付け or パスを指定）
```

または直接:
```
/sf-analyze docs/requirements/企画書.md
```

読み込ませるべき資料の例:
| 資料 | 効果 |
|---|---|
| 企画書・提案書 | プロジェクトの目的・スコープを Claude Code が理解する |
| ヒアリングメモ | 業務要件・現場の課題を反映した回答ができるようになる |
| 既存のER図・設計書 | 設計判断の精度が上がる |
| 用語集 | プロジェクト固有の用語を正しく扱える |

### Step A-3. /sf-analyze で組織解析

組織に認証済みの場合:
```
/sf-analyze
```

組織が空でも実行できる。「カスタマイズなし」「利用開始前」等のプロフィールが生成される。
読み込ませた資料がある場合、その内容も統合される。

生成されるファイル:
- `docs/overview/org-profile.md` — 組織プロフィール
- `docs/requirements/requirements.md` — 要件定義書（AS-IS / TO-BE）

### Step A-4. 生成資料のレビュー・修正

自動生成された資料を確認し、「推定」「要確認」の箇所を修正する。
特に以下を優先的に確認:
- 業種・事業内容の推定が合っているか
- 要件の優先順位が正しいか
- スコープ外の項目が含まれていないか

### Step A-5. Git初回push

```bash
git init
git add .
git commit -m "feat: プロジェクト初期セットアップ"
git remote add origin <GitHubリポジトリURL>
git push -u origin main
```

## 4. パターンB: 保守案件（既存組織の引き継ぎ）

既にカスタマイズされたSalesforce組織がある。引き継ぎ・機能追加・改修を行う案件。

### Step B-1. 組織認証 + メタデータ取得

`/setup-sf-project` で組織認証まで済ませた後、メタデータを取得する:

```
メタデータを取得しますか？ → はい
```

これでApex・LWC・Flow・オブジェクト定義等が `force-app/` に取り込まれる。

> **ポイント**: メタデータを取得することで、Claude Code が既存のコードを参照しながら作業できるようになる。保守案件ではこのステップが特に重要。

### Step B-2. /sf-analyze で現状把握

```
/sf-analyze
```

既存組織を解析し、以下を自動生成する:
- **組織プロフィール**: 業種推定・利用規模・データ構成・カスタマイズ度・技術的所見
- **要件定義書**: 現状（AS-IS）の整理・入力規則から読み取れるビジネスルール

**保守案件でこれが特に有用な理由:**
- 前任者のカスタマイズ意図を推定できる
- APIバージョンの古いApexクラス等の技術的負債が見える
- 入力規則・フローからビジネスルールを逆引きできる
- 用語集で「この会社での呼び方」がわかる

### Step B-3. 既存資料の読み込み

引き継ぎ資料・設計書・過去の議事録等があれば読み込む:

```
/save-doc
（ファイルを貼り付け）
```

保守案件で読み込ませるべき資料:
| 資料 | 効果 |
|---|---|
| **引き継ぎ資料** | 前任者の設計意図・注意事項を把握 |
| **過去の障害報告** | 地雷を事前に把握できる |
| **過去の設計書** | 既存設計との整合性を保てる |
| **運用手順書** | 本番作業の手順・禁止事項を把握 |
| **ER図** | データモデルの理解が深まる |

### Step B-4. /sf-analyze の差分更新（資料読み込み後）

資料を読み込んだ後、再度 `/sf-analyze` を実行すると:
- 読み込んだ資料の情報が組織プロフィール・要件定義書に統合される
- 「推定」だった箇所が資料の情報で補完される

```
/sf-analyze
```

### Step B-5. CLAUDE.md の記入

`/sf-analyze` で生成された情報をもとに、`CLAUDE.md` を記入:

特に保守案件で重要なセクション:
- **注意事項・地雷**: 前任者から引き継いだ注意点、過去の障害
- **過去の判断・決定事項**: 「なぜこの設計にしたか」の記録

```markdown
## 注意事項・地雷
<!-- 2026-04-03: AccountトリガーとXXXパッケージが競合。条件チェックが必要 -->
<!-- 2026-04-03: Opportunityの入力規則「VR_001」はA社案件でのみ有効化する -->

## 過去の判断・決定事項
<!-- 2026-04-03: 受注オブジェクトはOpportunityを流用。新規オブジェクトは作らない -->
```

> `/feedback` コマンドでも追記できる:
> ```
> /feedback AccountトリガーとXXXパッケージが競合。条件チェックが必要
> ```

### Step B-6. Git初回push

```bash
git init
git add .
git commit -m "feat: プロジェクト初期セットアップ（既存組織メタデータ含む）"
git remote add origin <GitHubリポジトリURL>
git push -u origin main
```

## 5. 共通手順: メンバー招待

### Step 5-1. GitHubリポジトリにメンバーを招待

GitHub → Settings → Collaborators → Add people

### Step 5-2. オンボーディング資料を共有

以下のドキュメントをメンバーに案内する:
- [オンボーディングガイド](onboarding.md) — 環境構築手順
- [テンプレート説明書](template-guide.md) — 機能リファレンス

### Step 5-3. メンバーの環境構築を確認

各メンバーが以下を完了していることを確認:
- [ ] リポジトリをクローンできた
- [ ] Salesforce組織に認証できた
- [ ] Claude Codeが起動し、プロジェクトルールが読み込まれている
- [ ] `/setup-mcp` でMCP連携を設定した（必要な場合）

## 付録: セットアップ後の最初の作業例

### 新規案件の場合

```
1. /sf-analyze の要件定義書をもとに、詳細設計を開始
   「要件FR-001の機能設計書を作って」

2. 最初の機能を実装
   /sf-implement 取引先登録画面のLWCコンポーネント

3. レビュー → デプロイ
   /sf-review → /sf-deploy dev環境にデプロイ
```

### 保守案件の場合

```
1. まず既存コードの理解
   「force-app/main/default/classes/ のApexクラスの一覧と役割を教えて」

2. 改修対象の特定
   /sf-review force-app/main/default/triggers/AccountTrigger.trigger

3. 改修の実装
   /sf-implement AccountTriggerにXXX条件のチェックを追加

4. 影響確認
   「この変更で影響を受けるフローやトリガーはある？」
```
