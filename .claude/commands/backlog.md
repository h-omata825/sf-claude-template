# /backlog [課題ID]

Backlog の保守案件チケットを読み込み、対応方針を分析・提案する。

## 引数

- `[課題ID]` — Backlog の課題ID（例: `DFA-123`）

## 手順

### 1. 課題の読み込み

指定された課題IDをBacklog MCPで取得する。

取得対象:
- 件名・説明・優先度・期限
- カテゴリ・マイルストーン
- コメント（最新10件）
- 添付ファイル一覧（参照のみ）

### 2. 組織情報との照合

課題の内容と、手元のメタデータ（`force-app/`・`docs/`）を照合する。

- 影響するオブジェクト・項目を特定
- 関連する Apex クラス・トリガー・フロー・LWC を特定
- 依存関係・影響範囲を整理

情報が不足している場合は `/sf-analyze` の実行を提案する。

### 3. 対応方針の提案

以下の形式でユーザに提示する:

```
## 課題: [課題ID] [件名]

### 要件・背景
[課題の内容をまとめる]

### 影響範囲
- オブジェクト: [一覧]
- コンポーネント: [一覧]

### 対応方針（案）

#### 案A: [方針名]
- 概要: ...
- メリット: ...
- デメリット: ...
- 工数: [小 / 中 / 大]

#### 案B: [方針名]（2つ以上ある場合）
- 概要: ...
- メリット: ...
- デメリット: ...
- 工数: [小 / 中 / 大]

### 推奨案
[根拠付きで推奨案を提示]

### 確認事項
[実装前にユーザに確認すべき点があれば列挙]
```

### 4. ユーザとの合意

提案後、ユーザと対応方針を確認し合意を得てから実装に移る。

実装は `salesforce-dev` エージェントが担当する。

## 使用例

```
/backlog DFA-123
/backlog GF-45
```

## 注意

- このコマンドは「分析・提案」まで。実装はユーザの承認後に別途依頼する
- 課題が見積未完了の場合は、見積後に再実行を推奨する
- 複数課題を同時に処理する場合は、1件ずつ実行する
