# /backlog [課題ID]

Backlog の保守案件チケットを読み込み、原因・影響範囲を徹底調査した上で対応方針を分析・提案する。

**【鉄則】**
- 推測で原因を断定しない。コードを読み、動作を確認し、根拠を示して結論を出す
- 「おそらく〜」「〜と思われる」で原因を確定させない
- 影響範囲は直接・間接の両方を網羅的に洗い出す

## 引数

- `[課題ID]` — Backlog の課題ID（例: `DFA-123`）

---

## Phase 1. 課題の読み込みと内容確認

Backlog MCP で課題を取得する。

取得対象:
- 件名・説明・優先度・期限・ステータス
- カテゴリ・マイルストーン
- コメント（全件）
- 添付ファイル一覧（参照のみ）

取得後、課題内容を**自分の言葉で要約**してユーザに確認する。
理解に誤りがあれば修正してから先に進む。

---

## Phase 1.5. 情報充足度の判定

課題内容の理解後、対応方針の検討に進めるかを判定する。

### 判定観点

- 課題が求めている**ゴール**（期待する動作・状態）
- バグの場合: **再現条件**
- 追加要望の場合: **何をどう変えるか**の具体性
- 対応に必要な**技術情報**（対象オブジェクト・機能・画面等）

### 3段階の判定と動き方

| 判定 | 状態 | 動き方 |
|---|---|---|
| **A: 明確** | 全て揃っている | そのまま Phase 2 へ |
| **B: 概ね推測可能** | 8割方分かるが一部曖昧 | **推測を明示して Phase 2 に進む**。前提が違えば軌道修正する |
| **C: 根本的に不明** | ゴール自体が分からない / 解釈が複数あり絞れない | 確認してから進む |

**原則: 確認で止まるより、推測込みで走らせて軌道修正する方が速い。**
判定Bが最も多いケースになる。止めるのはCの場合のみ。

### 判定B（推測込みで進む）の場合

Phase 2 の冒頭で以下を提示してから調査に入る:

```
⚠ 以下の前提で進めます。認識が違えば指摘してください。
- [推測した前提①]
- [推測した前提②]
```

ユーザが指摘しなければそのまま進行する。

### 判定C（確認が必要）の場合

```
情報不足を検知
  ↓
【Step 1】不足情報の特定
  何が分かれば対応方針を示せるのかを具体的にリストアップする。
  曖昧な依頼の場合は、考えられる解釈を列挙して「どれですか？」と絞り込む。
  ↓
【Step 2】ユーザへの確認
  不足情報をユーザに確認する。
  ユーザが把握している情報で解消できればそのまま Phase 2 へ進む。
  ↓
【Step 3】お客様への質問の構成（ユーザでも不明な場合）
  以下の形式で、お客様への質問案をユーザに提示する:

  ───────────────────────
  ## お客様への確認依頼（案）

  ### 背景
  [なぜこの質問が必要か。お客様に伝わる言葉で]

  ### 確認事項
  1. [具体的な質問。Yes/No or 選択肢で答えられる形が望ましい]
  2. ...

  ### この情報が得られると
  [何が可能になるか。「この回答により、○○の対応方針を確定できます」]
  ───────────────────────

  【重要】「この情報があれば解決できる」と提示する以上、
  それが得られた場合に確実に対応方針を示せることを保証すること。
  不確実な場合は正直に「方針の精度が上がる」等の表現にとどめる。
```

## Phase 2. 原因・影響の徹底調査

> **この Phase が最も重要。表面的な推測で原因を決めず、コードと設定を実際に読んで検証する。**

### 2-1. 課題種別の判断

- **バグ**: 既存の動作が期待と異なる
- **追加要望**: 新機能・仕様変更の要求
- **その他**: 設定変更・データ修正等

### 2-2. 関連コンポーネントの特定

`force-app/` および `docs/` を読み込み、以下を特定する:

- 関係するオブジェクト・項目
- 関係する Apex クラス・トリガー・フロー・LWC・権限セット等
- 呼び出し関係・依存関係（上流・下流）

複数の候補がある場合は全て列挙し、それぞれ根拠を示して絞り込む。

### 2-3. バグの場合: 原因の特定（多重検証）

1. **仮説を複数立てる** — 症状から考えられる原因を列挙
2. **コードで各仮説を検証** — 該当コードを実際に読んで確認
3. **根拠を明示** — 「〇〇ファイルの〇〇行目の処理が原因。なぜなら〜」の形式
4. **他の候補を排除** — 絞り込んだ理由を説明
5. **論理的に再検証** — 特定した原因で症状が説明できるか確認

### 2-4. 影響範囲の調査（直接・間接）

修正/追加が影響する可能性のある箇所を**網羅的に**洗い出す:

- 直接影響: 修正対象のファイル・設定
- 間接影響: 修正対象を参照・呼び出している箇所
- データ影響: レコードデータへの副作用の可能性
- 権限影響: 権限セット・プロファイルへの影響

---

## Phase 3. 対応方針の提案

以下の形式でユーザに提示する:

```
## 課題: [課題ID] [件名]

### 課題種別
[バグ / 追加要望 / その他]

### 要件・背景
[課題の内容・ユーザの期待する動作]

### 【バグの場合】原因特定結果

#### 特定した原因
[ファイル名・行番号・処理内容]

#### 根拠
[なぜこれが原因と断定できるか。コードの該当箇所を引用して説明]

#### 排除した候補と理由
[他に疑った箇所と、排除した理由]

### 影響範囲

#### 直接影響
- [ファイル/オブジェクト/項目名]: [内容]

#### 間接影響
- [参照元・呼び出し元]: [内容]

#### データ・権限への影響
[あれば記載。なければ「なし（根拠: 〜）」]

### 対応方針（案）

#### 案A: [方針名]
- 概要: ...
- メリット: ...
- デメリット: ...
- リスク: ...
- 工数: [小 / 中 / 大]

#### 案B: [方針名]（複数ある場合）
- ...

### 推奨案と根拠
[どの案を推奨するか。理由を明示]

### 実施前の確認事項
[実装前にユーザに確認すべき点を列挙]
```

---

## Phase 3.5. 対応方針のクロスレビュー（実装前）

対応方針の提案後、**実装に入る前に** `reviewer` の観点で方針自体をレビューする。

### レビュー観点
- 対応方針に**抜け漏れ**がないか（影響範囲の見落とし、考慮不足の権限・設定）
- 修正範囲が適切か（不要な変更が含まれていないか、必要な変更が漏れていないか）
- 実装後に発生しうる**副作用**を全て洗い出しているか
- **権限・セキュリティ**への影響を確認しているか:
  - 項目追加・変更 → FLS（項目レベルセキュリティ）の設定が必要か
  - オブジェクト変更 → CRUD権限の確認
  - Apex変更 → `with sharing` / FLSチェックの確認
  - フロー変更 → 実行ユーザの権限で動作するか
  - ページレイアウト → 必要なプロファイルに反映されるか

### レビュー結果
問題がある場合は Phase 3 の提案内容を修正してからユーザに提示する。
問題がなければそのまま Phase 4 へ。

---

## Phase 4. ユーザとの合意と実装準備

提案後、ユーザと対応方針を確認し合意を得る。

合意後、対応内容に応じて担当エージェントを選定する:

| 対応内容 | 担当エージェント |
|---|---|
| Apex / LWC / Flow / メタデータ実装 | `salesforce-dev` |
| 本番障害・デバッグログ解析・パフォーマンス問題 | `maintenance` |
| データ修正・データ移行・SOQL | `data-manager` |
| 外部API連携・Named Credentials | `integration-dev` |
| 設定変更のみ（権限・レイアウト等） | `salesforce-dev` |
| 複合的な対応 | 作業を分解し、各エージェントに割り当て |

選定したエージェントに以下のコンテキストを引き継ぐ:
- 課題内容の要約
- 確定した対応方針
- 影響範囲の調査結果
- 実施前の確認事項

---

## Phase 5. 実装

担当エージェントが対応方針に従い実装する。

### 実施ルール
- 修正前に対象ファイルを必ず読んで現状を把握する
- 修正は最小限にとどめる（無関係な箇所を変更しない）
- 変更内容を Before / After 形式でユーザに提示する

---

## Phase 6. テスト・検証（多層検証）

> **実装担当者として、人間が気づけない箇所も含めて全ての観点を網羅する。**
> **ユーザのエビデンス確認は最終確認であり、問題発見の場ではない。**

### 6-1. 実装レビュー（`reviewer` 観点）

実装完了直後に、変更内容を `reviewer` の基準でセルフレビューする。

**コード品質:**
- [ ] ガバナ制限に抵触しないか（SOQL/DML/CPU）
- [ ] バルク処理に対応しているか（200件以上）
- [ ] エラーハンドリングが実装されているか
- [ ] ハードコードがないか
- [ ] 命名規則に従っているか

**セキュリティ・権限（FLSの件のような凡ミスを絶対に防ぐ）:**
- [ ] 項目追加・変更 → **FLS（項目レベルセキュリティ）が設定されているか**
- [ ] オブジェクト変更 → CRUD権限が適切か
- [ ] Apex → `with sharing` が適切に設定されているか
- [ ] Apex → FLSチェック（`Security.stripInaccessible` 等）が必要な箇所にあるか
- [ ] フロー → 実行ユーザの権限で正しく動作するか
- [ ] ページレイアウト → 必要なプロファイルに反映されているか
- [ ] 権限セット → 必要な権限が付与されているか

**設定の整合性:**
- [ ] ページレイアウトに新規項目が追加されているか
- [ ] 入力規則と新しいロジックが矛盾していないか
- [ ] 既存の自動化（トリガー・フロー）と競合していないか

問題があれば Phase 5 に戻って修正してからテストに進む。

### 6-2. 機能テスト（`qa-engineer` 観点）

**直接テスト（課題の解決確認）:**
- 課題で報告された症状が解消されているか / 要望が実現されているか
- **実際に動作確認する**（Apex テスト実行 / 画面操作 / SOQL で結果確認）
- 根拠（コードの動作フロー + 実行結果）を示す

**影響範囲テスト:**
- Phase 2 で洗い出した**全ての**影響箇所について正常動作を確認
- 直接影響: 変更したファイル・設定の動作確認
- 間接影響: 変更箇所を参照・呼び出しているコンポーネントの動作確認
- 同一オブジェクトの他のトリガー・フロー・入力規則が正常に動作するか

**エッジケーステスト:**
- NULL / 空値 / 空リスト
- 大量データ（200件以上のバルク処理）
- 境界値（文字数上限・数値の最大最小）
- 権限の異なるユーザ（管理者 / 一般ユーザ / 権限なしユーザ）

**権限テスト（項目・オブジェクトを変更した場合は必須）:**
- 各プロファイルで対象画面を開いて、項目の表示・編集が正しいか
- FLSが設定されていない場合、画面でエラーが発生しないか
- レポート・リストビューで新規項目が参照できるか

### 6-3. Apex テスト実行（コードを変更した場合）

```bash
# 変更クラスのテスト実行
sf apex run test --target-org <alias> --class-names <テストクラス名> --result-format human --code-coverage

# 関連テスト全体の実行（影響範囲確認）
sf apex run test --target-org <alias> --test-level RunLocalTests --result-format human --code-coverage
```

- カバレッジ 90% 以上を確認
- 全テストがパスすることを確認
- フェイルしたテストがあれば原因を調査し、Phase 5 に戻る

### 6-4. テスト結果の報告

```
## テスト結果報告

### 実装レビュー（6-1）
| チェック項目 | 結果 | 備考 |
|---|---|---|
| ガバナ制限 | ✅ OK | |
| FLS設定 | ✅ OK | 全プロファイルに参照・編集権限を付与済み |
| ... | | |

### 機能テスト（6-2）
| 区分 | テスト項目 | 確認方法 | 結果 | 根拠 |
|---|---|---|---|---|
| 直接 | [項目] | [どう確認したか] | ✅ OK / ❌ NG | [根拠] |
| 影響範囲 | [対象] | [確認内容] | ✅ OK / ❌ NG | [根拠] |
| エッジケース | [項目] | [確認方法] | ✅ OK / ❌ NG | [根拠] |
| 権限 | [プロファイル名] | [確認内容] | ✅ OK / ❌ NG | [根拠] |

### Apexテスト（6-3）
- 実行テスト数: X
- パス: X / フェイル: X
- カバレッジ: X%

### 総合判定
[全項目OK → Phase 7 へ / NG項目あり → Phase 5 に戻る]
```

**NG項目がある場合は Phase 5 に戻って修正し、再度 Phase 6 を最初からやり直す。**
**全項目 OK になるまでユーザに報告しない。**

テスト結果報告後、ユーザにエビデンス取得を促す:
```
テスト完了。全項目OKです。
以下の画面でエビデンス（スクリーンショット）の取得をお願いします:
1. [確認すべき画面・操作の手順]
2. ...
```

## Phase 7. デプロイ・完了

### テスト環境へのデプロイ
1. デプロイ対象の資材を一覧化
2. デプロイ検証（`sf project deploy start --dry-run`）を実行
3. デプロイ実行
4. デプロイ後の動作確認

### 対応記録の更新
- `docs/decisions.md` に判断記録を追記（対応方針の選定理由・排除した案とその理由）
- 変更したオブジェクト・コード・設定に対応する `docs/` の資料を更新

### 本番デプロイ（ユーザが指示した場合のみ）
- 本番環境へのデプロイは**必ずユーザの明示的な指示**を待つ
- デプロイ手順・ロールバック手順を提示してから実行

---

## 使用例

```
/backlog DFA-123
/backlog GF-45
```

## 注意

- 各 Phase の終わりにユーザの確認・承認を得てから次に進む
- Phase 1.5 で情報不足を検知した場合は判定に従って対応する
- 実装前に必ず影響範囲を把握し、テストで検証する
- 本番デプロイは必ずユーザの明示的な指示が必要
- 複数課題を同時に処理しない。1件ずつ実行する
- 証跡用 Excel が必要な場合は `/backlog-report` を使う
