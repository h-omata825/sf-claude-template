# /backlog [課題ID] または [summary]

保守課題の対応を原因調査から完了まで一気通貫で実施する。

**`/backlog [課題ID]`** — 調査・実装・エビデンス取得まで完遂（メインワークフロー）
**`/backlog summary`** — `docs/logs/effort-log.md` の工数サマリーを出力

---

## 引数

- `[課題ID]` — Backlog の課題ID（例: `GF-327`）
- `summary` — 全課題の工数サマリーを出力して終了

---

## [summary] モードの動作

`docs/logs/effort-log.md` を読み込み、以下の形式でサマリーを出力する:

```
## 工数サマリー

### 全課題一覧
| 課題ID | 件名 | 種別 | 見込み（CC） | 見込み（非CC） | 実績（CC） | 削減効果 | 対応日 |

### 集計（実績記録済みのみ）
- 対応課題数 / CC使用合計 / CC未使用見込み合計 / 削減時間合計 / 平均削減率 / 見込み精度

### 種別内訳（バグ / 追加要望 / その他）
```

---

## [課題ID] モードの動作

> **【絶対ルール】**
> - 各Stepは必ず順番通りに実行する。前のStepが完了するまで次に進まない
> - 実装は Step 7 以降。それ以前に実装コードを書くことは禁止

---

## Step 0. 開始時刻を記録する

```bash
date "+%Y-%m-%d %H:%M:%S"
```

この時刻を会話の最後まで保持する。

---

## Step 1. 課題取得と内容確認

Backlog MCP で課題を取得する:
- 件名・説明・優先度・期限・ステータス・カテゴリ・マイルストーン・コメント（全件）

取得後、以下を実施する:

1. 課題内容を自分の言葉で要約してユーザに提示する
2. **コメントに要件変更・追加依頼がないか確認する**（「〇〇も対応してください」「〇〇に変更してください」等）  
   → 後付け要件が見つかった場合は、課題概要に「追加要件」として明示してからユーザに提示する

> **次に進む条件: ユーザが確認した後**

---

## Step 2. 対応記録ファイルの作成（選択式）

AskUserQuestion ツールで以下を表示する:

**質問**: 「対応記録・エビデンスxlsxを作成しますか？」

**選択肢**:
- label: "作成する（推奨）" description: "対応記録.xlsx と エビデンス.xlsx を作成して記録を残す"
- label: "作成しない" description: "ファイルなしで調査・対応のみ行う"

### 「作成しない」が選ばれた場合

xlsxは作成しない。Step 3 以降はチャットのみで調査・対応を進める。
工数記録（`docs/logs/effort-log.md`）は Step 5 で見込み・Step 12 で実績を直接テキスト追記する。

### 「作成する」が選ばれた場合

**2-1. フォルダパスの確定**

保存先フォルダを確認する（プロジェクトのCLAUDE.mdに `BACKLOG_REPORT_DIR` が定義されていればそれを使用する）。

フォルダパスをユーザに提示して確認を取る:
```
作業フォルダ: {BACKLOG_REPORT_DIR or デフォルトパス}/{課題ID}_{件名}\
このフォルダに対応記録.xlsx と エビデンス.xlsx を作成します。よろしいですか？
```

> **次に進む条件: ユーザが確認した後**

**2-2. xlsx の生成**

以下を実行する:
```bash
python scripts/python/backlog-xlsx/create_records.py \
  "{作業フォルダパス}" "{課題ID}" "{件名}" "{バグ/追加要望/その他}" "{優先度}" "{期限}" "{背景・要件の要約}"

python scripts/python/backlog-xlsx/create_evidence.py \
  "{作業フォルダパス}" "{課題ID}"
```

生成完了後、ユーザに報告する:
```
xlsx 生成完了
- {課題ID}_対応記録.xlsx（6シート）
- {課題ID}_エビデンス.xlsx（3シート）
保存先: {作業フォルダパス}
```

### xlsx 逐次更新ルール（「作成する」選択時のみ）

各 Step 完了時に対応する xlsx シートを更新する:

| Step | 更新対象 | 記録内容 |
|---|---|---|
| Step 1 | サマリー・経緯 | 課題サマリー欄 + タイムラインに「課題取得」追加 |
| Step 3 | 調査・影響範囲 | 仮説検証テーブルに調査結果を記録 |
| Step 4 | エビデンス.xlsx | テスト仕様シート + 実装前エビデンスシート |
| Step 5 | 対応方針 | 方針比較テーブル + 採用方針 |
| Step 7 | 対応内容 | Git hash・変更ファイル一覧・Before/After |
| Step 8 | テスト・検証記録 | テスト結果を全行記録 |
| Step 9 | エビデンス.xlsx | 実装後エビデンスシート |
| Step 10 | リリース・ロールバック | リリース対象一覧・ロールバック手順 |
| Step 12 | サマリー・経緯 | 完了時刻・実績工数・最終対応サマリー |

**ユーザとのやりとり・方針変更・追加修正が発生するたびに、タイムラインに行を追加する。**

---

## Step 3. 調査・影響範囲の特定

`force-app/` および `docs/` を読み込み、以下を実施する:

- 関連コンポーネント（オブジェクト・Apex・フロー・LWC・権限セット等）を特定
- バグの場合: 仮説を複数立て、コードを実際に読んで多重検証。根拠を示して原因を特定
- 追加要望の場合: 関連する既存実装を全て読んでから方針を検討
- 影響範囲（直接・間接・データ・権限）を網羅的に洗い出す

### デプロイ適否チェック

以下のいずれかに該当する場合、**コード実装・デプロイは不適切**と判断する。Step 5 の方針提案で「管理画面直接操作」として案内し、Step 7〜10（実装・デプロイ）をスキップする:

- 本番環境とリポジトリのメタデータが大きく乖離しており、デプロイすると本番の意図的な設定が上書きされるリスクがある
- 変更対象がページレイアウト・プロファイル・権限セット等「管理画面から直接編集する方が安全」なメタデータ
- コード変更が不要で、Salesforce 管理画面の設定変更だけで完結する

→ 該当する場合は Step 5 の方針欄に「実装不要・管理画面直接操作」と明記し、操作手順をユーザに案内する。

---

## Step 4. 実装前エビデンスの取得（可能であれば）

1. テスト仕様を検討して記録する（**必須**）
2. Playwright で撮影する前に組織の認証状態を確認する:
   ```bash
   sf org list
   ```
   - 認証が切れている場合は再認証を試みる:
     ```bash
     sf org login web --target-org <alias>
     ```
   - 再認証が失敗した場合（SSO・MFA等の制約）はスキップして Step 5 へ進む

3. 以下の条件を全て満たす場合のみ Playwright でスクリーンショットを撮影する:
   - Salesforce組織の URL にアクセスできる（ネットワーク制限なし）
   - ブラウザのセッションが有効（ログイン済み / SSO通過済み）
   - 対象画面が Lightning Experience の静的な画面（動的レンダリング完了後に要素が確定する）

   **スキップして Step 5 へ進む条件（いずれか1つ該当）:**
   - 組織URLへの接続がタイムアウト / アクセス不可
   - ログインセッションが切れており再認証が必要
   - Lightning の動的ローディングにより画面が白紙または要素未描画
   - 画面を開くには特定の操作手順（ウィザード・モーダル）が必要で再現が困難

   `保存先: {作業フォルダパス}\before_{連番}_{説明}.png`

---

## Step 5. 対応方針の提案とユーザ承認

以下の形式で提示する:

```
## 課題: [課題ID] [件名]

### 原因 / 要件
### 影響範囲
### 対応方針（案）
  案A★ [方針名] — 概要 / メリット / デメリット / リスク / 見込み工数（CC使用）: Xh
  案B [方針名]（複数ある場合）
### 推奨案と根拠
### 実施前の確認事項
```

工数見積もりも提示する（CC使用 Xh / CC未使用 Xh / 効率化 約X倍）。
`docs/logs/effort-log.md` に見込みを追記する。

> **次に進む条件: ユーザが方針を承認した後。承認なしに実装に進むことは禁止。**

---

## Step 6. 方針のクロスレビュー

実装前に以下を確認する:
- 影響範囲の見落としがないか
- FLS / CRUD / `with sharing` / ページレイアウト / 権限セットへの影響
- 副作用が全て洗い出されているか

問題があれば Step 5 の提案内容を修正してから次に進む。

---

## Step 7. 実装

担当エージェントを選定して実装する:

| 対応内容 | 担当エージェント |
|---|---|
| Apex / LWC / Flow / メタデータ | `sf-dev` |
| 本番障害・パフォーマンス問題 | `maintenance` |
| データ修正・移行 | `data-manager` |
| 外部API連携 | `integration-dev` |

ルール:
- 修正前に対象ファイルを必ず読む
- 修正は最小限にとどめる
- Before / After 形式でユーザに提示する

---

## Step 8. テスト・検証

### 8-1. 実装レビュー（reviewer 観点）

- [ ] ガバナ制限 / バルク処理 / エラーハンドリング / ハードコードなし
- [ ] FLS / CRUD / `with sharing` / ページレイアウト / 権限セット

### 8-2. 機能テスト（Playwright で実施）

- テスト用新規データを作成する（既存の本番・顧客データは使わない）
- ユーザが行う操作フローを通して実行する
- 変更した機能・同画面の既存機能・影響範囲の全てを確認する

### 8-3. Apex テスト（コード変更がある場合）

```bash
sf apex run test --target-org <alias> --class-names <テストクラス名> --result-format human --code-coverage
```

カバレッジ 90% 以上・全テストパスを確認する。

### 8-4. テスト結果報告

```
## テスト結果報告
| 区分 | テスト項目 | 確認方法 | 結果 | 根拠 |
NG項目があれば Step 7 に戻る。全項目 OK になるまでユーザに報告しない。
```

---

## Step 9. 実装後エビデンスの取得（可能であれば）

Step 4 と同じ手順（認証確認 → 必要なら `sf org login web` → 失敗したらスキップ）を適用する。

1. Step 4 のテスト仕様に従い実装後の画面を撮影する:
   `保存先: {作業フォルダパス}\after_{連番}_{説明}.png`

---

## Step 10. デプロイ

1. デプロイ対象の資材を一覧化する
2. デプロイ検証を実行する:
   ```bash
   sf project deploy start --dry-run --source-dir force-app
   ```
3. ユーザにデプロイ実行の確認を取る
4. デプロイを実行する
5. デプロイ後の動作確認を行う

本番デプロイはユーザの明示的な指示が必要。

---

## Step 11. ドキュメント更新

- `docs/decisions.md` に判断記録を追記する（方針の選定理由・排除した案）
- 変更したオブジェクト・コード・設定に対応する `docs/` の資料を更新する

---

## Step 12. 工数記録・完了処理

```bash
date "+%Y-%m-%d %H:%M:%S"
```

ユーザに経過時間・ブレーク時間を確認して実績工数を計算する。
`docs/logs/effort-log.md` の該当行に実績工数と削減効果を記録する。

完了サマリーを報告する:
```
## {課題ID} 対応完了

### 工数
| 区分 | 工数 |
| 見込み（CC） | Xh |
| 見込み（非CC） | Xh |
| 実績（CC） | Xh |
| 削減効果 | Xh 削減（X%） |

### 成果物（xlsx作成した場合）
- {課題ID}_対応記録.xlsx（6シート記録済み）
- {課題ID}_エビデンス.xlsx（テスト仕様 + エビデンス）

### 次のアクション
- [ ] 本番デプロイ（要ユーザ指示）
- [ ] お客様への完了報告
```

---

## 使用例

```
/backlog GF-327     # GF-327 の対応を実施
/backlog summary    # 全課題の工数サマリーを出力
```

## 注意

- 各 Step は必ず順番通りに実行する
- Step 5 のユーザ承認なしに実装に進むことは禁止
- 本番デプロイは必ずユーザの明示的な指示が必要
- xlsx を作成した場合、ユーザとのやりとりが発生するたびにタイムラインに追記する
