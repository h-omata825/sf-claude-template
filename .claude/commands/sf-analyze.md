description: "接続中のSalesforce組織を解析し、要件定義書・組織概要資料を自動生成する。既存の企画書・要件書があれば読み込んで統合する。"
---

salesforce-architectエージェントとして、接続中のSalesforce組織を解析し、要件定義資料を作成してください。

## ユーザー入力

$ARGUMENTS

上記の入力がある場合、以下のように解釈する:
- **ファイルパス**（.md, .txt, .xlsx, .pdf, .docx 等）→ 該当ファイルを読み込んで分析に統合する
- **フォルダパス** → フォルダ内の全ドキュメントを読み込む
- **テキスト** → プロジェクトの補足情報（背景・目的・要件のメモ等）として扱う
- **空（引数なし）** → 組織情報のみで解析する

複数指定された場合は全て読み込む。

---

## 実行手順

### Phase 1: 組織情報の自動収集

以下のコマンドを実行して組織の全体像を把握する。
**重要: `sf` コマンドはGit Bashのパス問題を回避するため、失敗した場合は以下の形式で実行する:**
```bash
"C:/Program Files/sf/client/bin/node.exe" "C:/Program Files/sf/client/bin/run.js" <サブコマンド> <引数>
```

#### 1-1. 組織基本情報
```bash
sf org display --json
```
→ 組織ID、インスタンスURL、APIバージョン、ユーザー名を取得

#### 1-2. カスタムオブジェクト一覧
```bash
sf sobject list -s custom
```

#### 1-3. 主要オブジェクトの項目構成
カスタムオブジェクトと主要標準オブジェクト（Account, Contact, Opportunity, Case, Lead）について:
```bash
sf sobject describe -s <オブジェクト名> --json
```
→ カスタム項目、リレーション、入力規則を抽出

#### 1-4. Apexクラス一覧
```bash
sf data query -q "SELECT Name, ApiVersion, Status, CreatedDate FROM ApexClass ORDER BY CreatedDate DESC" --json
```

#### 1-5. Flow一覧
```bash
sf data query -q "SELECT ApiName, ActiveVersionId, Description FROM FlowDefinitionView" --json
```

#### 1-6. レコード件数（主要オブジェクト）
```bash
sf data query -q "SELECT COUNT() FROM <オブジェクト名>" --json
```
→ Account, Contact, Opportunity, Case, Lead, 各カスタムオブジェクト

#### 1-7. 有効ユーザー数
```bash
sf data query -q "SELECT COUNT() FROM User WHERE IsActive = true" --json
```

#### 1-8. プロファイル一覧
```bash
sf data query -q "SELECT Name FROM Profile WHERE UserType = 'Standard'" --json
```

---

### Phase 2: 既存資料の読み込み

以下のフォルダに既存資料があるか確認する:
- `docs/requirements/` — 要件定義書・企画書・ヒアリングメモ
- `docs/design/` — 設計書
- `docs/` 直下 — その他ドキュメント

ファイルがあれば全て読み込み、内容を分析に統合する。

ファイルがない場合は、ユーザーに聞く:
```
プロジェクトの企画書・要件書・ヒアリングメモなどはありますか？
  - ファイルパスを指定してください（複数可）
  - なければ「なし」と入力してください（組織情報のみで解析します）
```

---

### Phase 3: 解析・資料生成

収集した情報をもとに、以下の資料を `docs/requirements/` に生成する。

#### 3-1. 組織概要書（org-overview.md）

```markdown
# 組織概要書

**生成日**: YYYY-MM-DD
**組織**: <組織ID> | <インスタンスURL>
**APIバージョン**: <バージョン>

---

## 組織プロフィール

| 項目 | 内容 |
|---|---|
| 推定業種 | （カスタムオブジェクト・項目名から推定） |
| Salesforceの利用目的 | （SFA / サービス / マーケティング / カスタムアプリ等） |
| 利用規模 | 有効ユーザー数: X名 |
| カスタマイズ度 | （低/中/高 — カスタムオブジェクト数・Apex数から判定） |

---

## データ構成

### オブジェクト一覧
| オブジェクト | 種別 | レコード数 | カスタム項目数 | 主なリレーション |
|---|---|---|---|---|

### データモデル（ER図）
```mermaid
erDiagram
    （主要オブジェクトのリレーションを図示）
```

---

## カスタマイズ構成

### Apex クラス
| クラス名 | APIバージョン | 作成日 | 推定用途 |
|---|---|---|---|

### フロー
| フロー名 | 種別 | 状態 | 説明 |
|---|---|---|---|

### 入力規則・自動化
| オブジェクト | 種別 | 名前 | 概要 |
|---|---|---|---|

---

## セキュリティ構成

### プロファイル一覧
| プロファイル名 | 推定ロール |
|---|---|

---

## 所見・推奨事項
- （組織の状態から見える課題・改善点）
- （技術的負債の兆候があれば指摘）
- （標準機能への置き換え候補があれば提案）
```

#### 3-2. 要件定義書（requirements.md）— 既存資料がある場合

```markdown
# 要件定義書

**プロジェクト名**: 
**作成日**: YYYY-MM-DD
**バージョン**: v1.0

---

## プロジェクト概要
（既存資料 + 組織情報から統合して記述）

## ビジネス要件
### 背景・課題
### 目的・ゴール
### スコープ

## 機能要件
（組織のカスタマイズ状況と要件書の内容を突き合わせて整理）

| # | 要件 | 優先度 | 実現方式（標準/カスタム） | 備考 |
|---|---|---|---|---|

## 非機能要件
| カテゴリ | 要件 | 詳細 |
|---|---|---|

## 現行システムとの差分
（AS-IS: 組織の現状 vs TO-BE: 要件書の目標）

## 未解決事項
| # | 質問・課題 | 担当 | 期限 |
|---|---|---|---|

## 前提条件・制約
```

### Phase 4: 報告

生成したファイルの一覧と概要をユーザーに報告する。
追加で深掘りしたい箇所があるか確認する。
