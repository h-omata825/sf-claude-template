description: "機能別の設計書を作成・読み込み・統合する。既存設計書のインポート、保守案件の全機能把握にも対応。"
---

salesforce-architectエージェントとして、機能別の設計書を作成してください。

## ユーザー入力

$ARGUMENTS

上記の入力がある場合、以下のように解釈する:
- **要件番号**（FR-001, FR-002 等）→ 該当要件の設計書を作成する
- **機能名**（「商談管理」「ケース自動振り分け」等）→ 該当機能の設計書を作成する
- **ファイルパス** → 既存設計書を読み込んで統合・標準化する
- **フォルダパス** → フォルダ内の設計書を一括読み込み（バルクインポート）
- **「インポート」+パス** → 保守案件導入モード。既存資料を一括で標準化して取り込む
- **空（引数なし）** → 要件定義書の全機能要件から設計書の作成計画を提示する

複数指定された場合は全て処理する。

### ファイル形式と読み込み方法
| 形式 | 方法 |
|---|---|
| .md, .txt, .csv, .json | Read ツールで直接読み込み |
| .pdf | Read ツールで読み込み（1回20ページまで。大きいPDFはページ指定で分割読み込み） |
| .xlsx | **Python で自動変換して読み込み**（下記の変換手順を参照） |
| .docx | **Python で自動変換して読み込み**（下記の変換手順を参照） |

### Excel / Word ファイルの自動変換手順

Claude Code は .xlsx / .docx を直接読めないため、Python で中間ファイルに変換してから読み込む。

#### .xlsx（Excel）の場合
```bash
python -c "
import pandas as pd
import sys

file_path = sys.argv[1]
xl = pd.ExcelFile(file_path)

for sheet_name in xl.sheet_names:
    df = pd.read_excel(xl, sheet_name=sheet_name)
    print(f'=== シート: {sheet_name} ===')
    print(df.to_markdown(index=False))
    print()
" "<ファイルパス>"
```
→ 出力されたテキストを分析に使用する。シートが複数ある場合は全シートを処理する。

#### .docx（Word）の場合
まず `python-docx` の有無を確認する:
```bash
python -c "import docx; print('OK')" 2>/dev/null || pip install python-docx
```
```bash
python -c "
import docx
import sys

doc = docx.Document(sys.argv[1])
for para in doc.paragraphs:
    print(para.text)
for table in doc.tables:
    for row in table.rows:
        print('| ' + ' | '.join(cell.text for cell in row.cells) + ' |')
" "<ファイルパス>"
```

#### フォルダ指定・バルクインポート時
フォルダ内の .xlsx / .docx も自動変換して処理対象に含める。

## 設計書のフォルダ構成

```
docs/design/
├── apex/           # Apexクラス・トリガーの設計書
├── flow/           # フロー（Screen / Record-Triggered / Scheduled / Autolaunched）
├── batch/          # バッチ Apex・スケジュールジョブ
├── lwc/            # Lightning Web Components
├── integration/    # 外部連携（REST/SOAP API, Platform Event 等）
├── config/         # 宣言的設定（入力規則, 数式, ページレイアウト, レコードタイプ等）
└── _index.md       # 全設計書のインデックス（自動生成）
```

### フォルダ振り分けルール
設計書の「実現方式」に応じて適切なフォルダに配置する:
- Apex クラス・トリガー → `apex/`
- フロー全般 → `flow/`
- バッチ・スケジュールジョブ → `batch/`
- LWC・Aura → `lwc/`
- 外部システム連携 → `integration/`
- 標準機能の設定（入力規則、数式、レイアウト等） → `config/`
- 複数種別にまたがる場合 → 主要な実現方式のフォルダに配置し、関連フォルダからシンボリックリンクではなく「関連設計書」セクションで相互参照

## Phase 0: コンテキスト読み込み

**以下のファイルを必ず読み込んでから作業を開始する:**

### 必須コンテキスト
| ファイル | パス | 用途 |
|---|---|---|

```

### 既存設計書
- `docs/design/` 配下の全サブフォルダ内のファイルを読み込む
- 既存設計書との重複・矛盾がないか確認する
- `_index.md` があれば現在の設計書一覧を把握する

### 組織メタデータ（必要に応じて）
設計対象の機能に関連するオブジェクト・項目の詳細情報が必要な場合:
```bash
sf sobject describe -s <オブジェクト名> --json
```

**重要: `sf` コマンドはGit Bashのパス問題を回避するため、失敗した場合は以下の形式で実行する:**
```bash
"C:/Program Files/sf/client/bin/node.exe" "C:/Program Files/sf/client/bin/run.js" <サブコマンド> <引数>
```

## Phase 1: 実行モード判定

### モード A: 設計計画の提示（引数なし）

要件定義書の機能要件一覧を読み、設計書の作成計画を提示する:

```
## 設計書 作成計画

要件定義書から以下の機能要件を検出しました。
設計書を作成する対象を指定してください（番号 or 機能名）。

| # | 要件 | フォルダ | 設計書 | 状態 |
|---|---|---|---|---|
| FR-001 | 商談管理機能 | apex/ | FR-001_opportunity-management.md | 未作成 |
| FR-002 | ケース自動振り分け | flow/ | FR-002_case-routing.md | 作成済み（v1.0） |
| FR-003 | 請求データ連携 | integration/ | — | 未作成 |

作成する要件番号を入力してください（複数可、例: FR-001 FR-003）

### モード B: 指定機能の設計書作成（要件番号 or 機能名指定）

指定された機能の設計書を生成する。実現方式に基づいてフォルダを自動判定する。

### モード C: 既存設計書の読み込み・統合（ファイルパス指定）

外部の設計書を読み込み、プロジェクトの標準フォーマットに変換・統合する。

### モード D: バルクインポート（「インポート」+フォルダパス / 大量ファイル指定）

**保守案件の導入時を想定**。既存の設計資料を一括で読み込み、標準テンプレートに変換・整理する。

## Phase 3: 既存設計書の読み込み・統合（モード C）

外部の設計書ファイルが指定された場合:

1. ファイルを読み込む（.md, .txt, .csv, .pdf は直接読み込み。.xlsx/.docx は Python で自動変換して読み込み）
2. 内容を分析し、以下を判定する:
   - **実現方式** → 配置先フォルダを決定（apex/ flow/ batch/ etc.）
   - **対応する要件番号** → 要件定義書と突き合わせ（紐づかない場合は MISC-XXX）
3. プロジェクトの標準テンプレートに変換する
4. 元の資料にあった情報で、テンプレートに該当セクションがない場合は「補足情報」セクションとして末尾に追加
5. 適切なサブフォルダに保存する

### 変換時のルール
- 元の資料の情報量を減らさない（テンプレートに収まらない情報は補足として残す）
- 用語は org-profile.md の用語集に合わせる
- 推定で補完した部分は「推定」と明記する
- 元ファイルのパスを設計書のメタ情報に記録する（トレーサビリティ）

## Phase 3D: バルクインポート（モード D — 保守案件導入用）

**保守案件でプロジェクトに参画した際、既存の設計資料を一括で取り込むモード。**

### 実行手順

1. **指定フォルダ内の全ファイルをスキャンする**
   - 対応形式: .md, .txt, .csv, .pdf, .xlsx, .docx（.xlsx/.docx は Python で自動変換して読み込み）
   - サブフォルダも再帰的にスキャン

2. **各ファイルの内容を分析し、分類する**
   - ファイル名・内容から実現方式（Apex/Flow/Batch/LWC/Integration/Config）を判定
   - 対応する要件番号を推定（紐づけできない場合は MISC-XXX）

3. **インポート計画をユーザーに提示する**
   ```
   ## バルクインポート計画

   以下のファイルを検出しました。分類結果を確認してください。

   | # | 元ファイル | 種別（推定） | 配置先 | 要件番号 |
   |---|---|---|---|---|
   | 1 | 商談管理設計書.xlsx | Apex | apex/FR-001_opportunity-management.md | FR-001 |
   | 2 | ケース振り分けフロー.pdf | Flow | flow/FR-002_case-routing.md | FR-002 |
   | 3 | 日次バッチ仕様.docx | Batch | batch/FR-005_daily-summary-batch.md | FR-005 |
   | 4 | API連携設計.md | Integration | integration/FR-010_invoice-api.md | FR-010 |
   | 5 | 不明な資料.txt | — | 判定不能 | — |

   修正がある場合は指示してください。
   問題なければ「実行」と入力してください。
   ```

4. **承認後、全ファイルを一括変換する**
   - 各ファイルを標準テンプレートに変換
   - 適切なサブフォルダに配置
   - 情報が不足しているセクションは「要確認」として残す

5. **インデックスファイルを生成する**（`_index.md`）

6. **変換結果を報告する**
   ```
   ## バルクインポート完了

   | 結果 | 件数 |
   |---|---|
   | 変換成功 | X件 |
   | 判定不能（手動確認要） | X件 |
   | エラー | X件 |

   以下のファイルが生成されました:
   - docs/design/apex/ — X件
   - docs/design/flow/ — X件
   - docs/design/batch/ — X件
   ...

   判定不能だったファイル:
   - （ファイル名と理由）

   次のアクション:
   - 各設計書の「要確認」箇所を埋める
   - 判定不能ファイルを手動で分類する
   - `/sf-analyze` で組織情報と突き合わせる
   ```

### バルクインポート時の注意
- 大量ファイルの場合、1回のコマンド実行でコンテキストが圧迫される可能性がある。その場合は「次の10件を処理します」のように分割して処理する
- 元ファイルは削除しない（変換元のトレーサビリティを保つため）
- 同名ファイルが既に存在する場合は上書きせず、ユーザーに確認する

## Phase 5: 差分更新

既存の設計書ファイルが存在する場合:

1. 既存ファイルを読み込む
2. 組織メタデータの変更を検出（オブジェクト・項目の追加/変更/削除）
3. 要件定義書の変更を検出（要件の追加/変更）
4. 手動で修正された内容は保持する
5. バージョン番号を更新する
6. `_index.md` を更新する

## 完了後: CLAUDE.md の自動更新

設計書の作成・統合が完了したら、ルートの `CLAUDE.md` を読み込み、以下を更新する。
**手動記入済みの内容は上書きしない。空欄・プレースホルダーのみ更新する。**

- **設計で確定した命名規則**（オブジェクト・項目・クラスのプレフィックス等）を命名規則セクションに追記
- **新たに判明した制約・注意事項**（設計上の制約・業務ルール等）を注意事項セクションに追記候補として提示し、ユーザーに確認してから追記
- **主要カスタムオブジェクト**（設計で新規追加されたオブジェクトがあれば）を追記
