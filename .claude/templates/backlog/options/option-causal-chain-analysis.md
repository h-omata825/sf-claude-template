# option-causal-chain-analysis

## 何をするか

症状 → 直接原因 → 根本原因の連鎖を図示する。中間層を飛ばして「根本原因を修正した」と思い込む誤りを防ぐ。

## 実行手順

1. 症状を起点にして因果連鎖を遡る:
   ```
   症状: {ユーザーが観察した問題}
     ↑ なぜ？
   直接原因 1: {コード・設定・データ上の直近の原因}
     ↑ なぜ？
   直接原因 2: {さらに一段階遡った原因}
     ↑ なぜ？
   根本原因: {これ以上遡れない、設計・要件・データ・仕様の問題}
   ```
2. 各「なぜ？」をコード・メタデータ・データで実際に確認する（推測で進めない）
3. 修正案が「直接原因 1」を修正するのか「根本原因」を修正するのかを明示する:
   - 直接原因の修正 → 場当たり対応。再発リスクあり
   - 根本原因の修正 → 抜本対応。影響範囲が広い可能性
4. 適切な修正レベルを判断する（option-conservative-vs-progressive での比較を推奨する場合も）

## 出力

investigation.md「根本原因」セクションに追記:

## 因果連鎖分析

```
症状: {ユーザー観察事象}
  ↑ なぜ？
直接原因: {コード/設定上の直接原因}（確認済み: {根拠}）
  ↑ なぜ？
根本原因: {設計/要件/データの問題}（確認済み: {根拠}）
```

今回の修正レベル: 直接原因（場当たり）/ 根本原因（抜本） / その中間
