# option-knowledge-extraction

## 何をするか

今回の調査・対応で得た知見を docs/ に残す。次の担当者が同種の問題に直面した時に活用できる知識を蓄積する。

## 実行手順

1. 今回の課題対応を振り返り、「次の人に役立つ知見」を列挙する:
   - 今回判明した設計・実装の制約（「この処理は〇〇の理由でこうしないと動かない」）
   - Salesforce の仕様で注意が必要な挙動
   - 回避策・ベストプラクティス
   - 再発防止のための注意点
   - 想定と異なった動作・調査で判明した意外な事実

2. 既存の docs/ 構成を確認する:
   - `docs/decisions/` — 設計決定の記録（decisions.md）
   - `docs/overview/` — 業務フロー・設計の概要

3. 知見を適切な場所に記録する:

   **設計決定として記録する場合** (`docs/decisions/{issueID}-{topic}.md` または `decisions.md` への追記):
   ```markdown
   ## {設計決定のタイトル}
   日付: {YYYY-MM-DD}
   課題: {issueID}
   決定内容: {何をどう決めたか}
   理由: {なぜそうしたか}
   代替案: {検討したが採用しなかった案}
   注意点: {将来変更する際の注意}
   ```

   **業務知識として記録する場合** (`docs/overview/` 等への追記):
   ```markdown
   ## {業務ルール・制約}
   判明日: {YYYY-MM-DD}
   内容: {業務的な制約・ルール}
   影響: {どの機能に影響するか}
   ```

4. 記録したことを test-report.md に記録する

## 出力

`docs/` の適切なファイルに追記 + test-report.md に追記:

## 知見抽出・docs 更新

- 記録先: {ファイルパス}
- 記録内容: {知見の要約}
