# option-symptom-reverification

## 何をするか

報告された症状の前提（固有名詞・実データ値・操作の実在性）を確認する。コード調査・原因仮説の前に事実関係を検証することで hallucination を防ぐ。

## 実行手順

1. 課題で報告されている症状の要素を抽出する:
   - 画面名・ボタン名・フィールド名（SF UI 上の表示ラベル）
   - 操作手順（どの順序で何をした）
   - 結果（何が起きた / 起きなかった）
2. 各要素の実在を確認する:
   - **UI 要素**: Salesforce 公式ドキュメントまたは Sandbox で当該画面を確認
   - **フィールド名・API 名**: `field-meta.xml` または `sf sobject describe` で確認
   - **操作手順**: Sandbox で同じ操作を実際に試みる（再現できるか）
3. 以下のパターンを特に疑う:
   - 「〇〇というボタン」が実際には存在しない UI 要素でないか
   - 「〇〇項目」が実際には別の API 名を持っていないか
   - データ側の問題（レコードの状態・フラグ値）が原因でないか
4. 確認結果を記録する:
   - 実在確認済み → 続行
   - 実在しない / 表記違い → 課題の前提を修正して investigation.md に記録

## 出力

investigation.md「根本原因 / 要件の本質」セクションの冒頭に追記:

**症状前提確認**:
- 〇〇（UI 要素名）: 実在確認済み / 表記違い（正しくは △△）/ 実在しない
- 〇〇（操作手順）: Sandbox で再現確認済み / 再現せず（条件: ...）
