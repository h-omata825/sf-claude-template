# option-rollback-strategy

## 何をするか

対応方針の段階でロールバック戦略を事前設計する。本番リリース後に問題が発生した場合の復旧手順を事前に確立する。

## 実行手順

1. 今回の変更内容から「ロールバックが難しい変更」を特定する:
   - データ変更・マイグレーションを含む
   - オブジェクト構造変更（フィールド追加・削除・型変更）
   - スキーマ変更（オブジェクト追加・削除）
2. 各変更に対するロールバック手順を設計する:

   **コード変更**:
   - git revert / 旧バージョン再デプロイ
   - 旧バージョンのデプロイ所要時間の見積もり

   **設定変更**（管理画面操作）:
   - 変更前の設定値を事前記録（スクリーンショット・メモ）
   - 手動での設定戻し手順

   **データ変更**:
   - バックアップ取得タイミング（変更前の Data Export / Data Loader 出力）
   - 戻し手順（Data Loader でのインポート）
   - 既に変更後の操作が行われた場合の対応

3. ロールバック判断基準を定義する:
   - どんな症状が出たらロールバックを実行するか
   - 誰が判断・実行するか

## 出力
