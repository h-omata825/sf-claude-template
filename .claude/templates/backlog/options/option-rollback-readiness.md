# option-rollback-readiness

## 何をするか

本番リリース前にロールバック手順を最終確認する。実際に問題が発生した場合に迷わず戻せる状態を確認する。

## 実行手順

1. 今回の変更内容に対するロールバック手順を確認する（approach-plan.md の「ロールバック戦略」またはリリース手順書）
2. 以下を確認する:

   **コード変更のロールバック**:
   - `git log --oneline -10` で今回の変更コミットを確認する
   - ロールバックコマンドを確認する: `git revert {コミットハッシュ}` または `git reset --hard {前のコミット}`
   - 再デプロイの所要時間を見積もる（Metadata API デプロイ: 通常 1〜10 分）

   **データ変更のロールバック**:
   - データ移行を行った場合: バックアップ CSV が存在するか確認する
   - Data Loader での戻し手順が準備されているか確認する

   **設定変更のロールバック**:
   - 管理画面操作を行った場合: 変更前の設定値がメモ・スクリーンショットで記録されているか確認する

3. ロールバック判断基準を確認する:
   - どんな症状が出たらロールバックするか（事前に決めておく）
   - 誰が判断するか・誰が実行するか

## 出力

リリース手順書（または test-report.md）に追記:

## ロールバック最終確認

- コードロールバック手順: `git revert {コミットハッシュ}` + 再デプロイ（所要: {N} 分）
- データロールバック: バックアップ有 / 不要
- 設定ロールバック: 変更前設定記録済み / 不要
- ロールバック判断基準: {どんな状態で判断するか}
- 担当: 判断 {氏名} / 実行 {氏名}
- 判定: ロールバック手順確認済み・リリース可
