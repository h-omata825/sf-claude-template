## 品質基準

### Apex — バルク処理パターン（必須）
```apex
trigger AccountTrigger on Account (before insert, before update) {
    AccountTriggerHandler.handle(Trigger.new, Trigger.oldMap);
}

public with sharing class AccountTriggerHandler {
    public static void handle(List<Account> newRecords, Map<Id, Account> oldMap) {
        // SOQLはループ外
        List<Account> targets = [SELECT Id, Name FROM Account WHERE ...];
        List<Account> toUpdate = new List<Account>();
        for (Account acc : newRecords) {
            // ループ内にDML・SOQL禁止
            toUpdate.add(new Account(Id = acc.Id, ...));
        }
        if (!toUpdate.isEmpty()) {
            Database.update(toUpdate, false);
        }
    }
}
```

詳細なコーディング規約・テスト要件・種別選定は「メタデータ種別ごとの振る舞い > Apex」を参照。

```bash

sf project deploy start --manifest manifest/package.xml --target-org project-dev --test-level RunLocalTests

# デプロイ状況確認

# Apexテスト実行
sf apex run test --target-org project-dev --test-level RunLocalTests --result-format human --code-coverage

# 特定クラスのテスト実行
sf apex run test --target-org project-dev --class-names MyClassTest --result-format human

# 匿名Apex実行
sf apex run --target-org project-dev --file scripts/apex/yourScript.apex

# 組織一覧
sf org list --all
```

以下は「〜作って」「〜変更して」等の指示を受けたときの、種別ごとの詳細な動き方を定義する。

---

### 項目（Custom Fields）

#### 作成時

1. **重複チェック**: `docs/catalog/` で同じ用途・似た名前の項目が既にないか確認。重複の疑いがあれば「既にXXX項目がありますが、別途必要ですか？」と確認
2. **命名**: プロジェクト `CLAUDE.md` の命名規則 → `docs/overview/org-profile.md` の用語集 → 一般的SF慣例の順で適用
3. **既存自動化の影響確認**: 対象オブジェクトに存在する自動化を全て把握する
   - `force-app/` から: トリガー・フロー・入力規則・数式項目・ロールアップ集計
   - `docs/design/` から: 設計書に記載された自動化ロジック
   - 新規項目が既存の入力規則・数式項目・フローの条件に影響しないか確認
   - 影響がある場合は「この項目の追加により、既存の○○に影響があります」と報告
4. **データ型選定**: 用途に応じて最適なデータ型を選定。迷う場合は選択肢と理由を提示して確認

**データ型ごとの考慮:**

| データ型 | デフォルト | 確認事項 |
|---|---|---|
| テキスト | 255文字 | 用途に応じて短縮を提案（「住所なら255、郵便番号なら10で十分では？」） |
| テキストエリア | — | 長文が必要か → テキストエリア(長い) / リッチテキストを提案 |
| 選択リスト | 制限あり | 値一覧を提示。`docs/data/master-data.md` に既存値があれば参照。レコードタイプがある場合はどのRTにどの値を割り当てるか確認 |
| 複数選択リスト | — | 本当に複数選択が必要か確認（レポート・SOQLでの扱いが複雑になる旨を警告） |
| 数式 | — | コンパイルサイズに注意。代替案（ロールアップ集計・フロー・Apex）との比較を提示。参照先の項目が変わるとエラーになるリスクを説明 |
| 参照関係 | — | 関連先オブジェクト確認。削除時の動作（参照クリア / 削除不可）を確認 |
| 主従関係 | — | **OWD・共有モデルへの影響を警告**。ロールアップ集計が使える旨に言及。既に主従関係がある場合は2つ目の制限を説明 |
| 通貨 / 数値 | 小数0桁 | 小数桁数を確認。マルチ通貨組織の場合は通貨型を推奨 |
| 日付 | Date | DateTime が必要か確認（タイムゾーンの影響を説明） |
| チェックボックス | false | デフォルト値を確認 |

**関連設定の確認:**

| 設定 | 動作 |
|---|---|
| FLS（項目レベルセキュリティ） | **特に指示がなければ確認せず自動設定する。** 組織のプロファイル構成・項目の用途・オブジェクトのOWDから判断し、必要なプロファイルに参照・編集権限を付与する（判断基準は下記参照）。指示がある場合はそれに従う |
| ページレイアウト | 「どのページレイアウトに追加しますか？（全て / 指定 / 追加しない）」 |
| コンパクトレイアウト | モバイル表示に関わる重要項目なら追加を提案 |
| 項目履歴管理 | 監査・追跡が必要な項目なら「項目履歴の追跡を有効にしますか？」と確認（上限20項目/オブジェクト） |
| 外部ID | データ移行・外部連携用途なら設定を提案 |
| ヘルプテキスト | 項目の用途が明確な場合はヘルプテキストを提案 |

**FLS自動判断の基準:**

以下の順序で判断する。判断に使った根拠と設定内容を作業結果に含めること。

1. **`docs/catalog/` を参照**: 同一オブジェクトの既存カスタム項目のFLS設定パターンを確認し、それに合わせる（最も信頼性が高い）
2. **項目の用途から判断**: カタログに情報がない場合、以下を目安にする
   - 一般業務項目（入力・表示用） → 全プロファイルに参照・編集可
   - 管理者・システム用項目（内部フラグ・連携ID等） → システム管理者のみ編集可、他は参照のみ or 非表示
   - 機密項目（給与・個人情報等） → 最小権限。必要なプロファイルのみに付与
3. **判断がつかない場合** → 全プロファイルに参照・編集可（安全側に倒す）

**上限チェック:**
- カスタム項目数が上限に近くないか確認（標準: 500、Enterprise以上: 800）
- カスタムインデックスが必要な場合（大量データ + SOQL条件で使用）はSalesforceサポートへの申請を案内

#### 変更時

1. **影響調査を必ず実施**してから変更する。以下を検索:
   - **Apex**: SOQL内の項目参照、コード内の直接参照（`obj.Field__c`）
   - **フロー**: 項目参照、条件分岐での使用
   - **入力規則**: 数式内の参照
   - **数式項目**: 他の数式からの参照
   - **レポート**: フィルター・グルーピング・列での使用
   - **ページレイアウト**: 配置状況
   - **ワークフロー / プロセスビルダー**: レガシー自動化での参照
2. 結果を一覧で提示: 「この項目は以下で使用されています: ...」
3. **型変更は特に警告**: データロスのリスク（例: テキスト→数値でデータ消失）を明示

#### 削除時

- 上記の影響調査に加えて「削除すると復元できません（ごみ箱に15日間保持後に完全削除）。本当に削除しますか？」と確認
- 削除ではなく「未使用」ページレイアウトセクションへの移動を代替案として提案

### オブジェクト（Custom Objects）

#### 作成時

1. **目的の確認**: 何を管理するオブジェクトか明確にする。既存の標準/カスタムオブジェクトで代替できないか `docs/catalog/` で確認
2. **命名**: プロジェクト `CLAUDE.md` の命名規則 → 用語集 → 一般慣例。ラベル・API名・複数形ラベルをセットで提示
3. **データモデル上の位置づけ**: `docs/catalog/_data-model.md` があれば、どこに位置するか確認

**基本設定の確認:**

| 設定 | デフォルト | 確認事項 |
|---|---|---|
| レコード名 | テキスト（Name） | 自動採番にするか確認。自動採番の場合は表示形式（例: `INV-{0000}`）と開始番号 |
| 説明 | — | オブジェクトの目的を簡潔に記載（必須ではないが推奨） |
| Allow Reports | 有効 | — |
| Allow Activities | 無効 | 活動（ToDo・行動）を紐づけるか確認 |
| Track Field History | 無効 | 監査要件があるか確認。後から有効化可能だが既存データの履歴は取れない |
| Allow in Chatter Groups | — | 通常はデフォルトのまま |
| Allow Search | 有効 | — |

**関連設定（オブジェクト作成後に順番に確認）:**

| 設定 | 確認内容 |
|---|---|
| **OWD（組織の共有設定）** | Private / Public Read Only / Public Read/Write のどれか。「このデータは全員が見れていいですか？それとも所有者ベースでアクセス制御が必要ですか？」と確認 |
| **タブ** | 作成するか確認。作成する場合はタブスタイル（アイコン）を提案。どのアプリケーションに表示するか確認 |
| **ページレイアウト** | デフォルトレイアウトの構成を提案。セクション構成・関連リスト・アクションボタン |
| **コンパクトレイアウト** | モバイル・ハイライトパネル用の主要項目（3〜4項目）を提案 |
| **レコードタイプ** | 業務上区分が必要か確認。「同じオブジェクトで異なるページレイアウト・選択リスト値が必要な場面はありますか？」 |
| **リレーション** | 関連オブジェクトとの関係を確認。主従 vs 参照の選定理由を説明（主従: カスケード削除・ロールアップ可 / 参照: 独立性維持） |
| **権限セット** | 「どの権限セット/プロファイルにCRUD権限を設定しますか？」と確認。自動では変更しない |

**設計上の注意点を自動で提示:**
- 主従関係を使う場合 → OWD・共有モデルが親オブジェクトに従属する旨を説明
- 大量データが見込まれる場合 → インデックス戦略・アーカイブ方針に言及
- 外部システム連携がある場合 → 外部ID項目の作成を提案

#### 変更時

1. **OWD変更は高リスク**: 共有再計算が走り、大規模組織では時間がかかる。影響範囲を説明した上でユーザー確認
2. **レコードタイプの追加/変更**: 既存レコードへの影響、ページレイアウト割り当て、選択リスト値の割り当てを確認
3. **ラベル変更**: API名は変わらないがUI上の表示が変わる。レポート名・ダッシュボード・ヘルプテキスト等への影響を確認

#### 削除時

- **全データが消失する旨を強く警告**（ごみ箱経由だが、関連レコードも連鎖削除される可能性）
- 関連する全メタデータ（項目、フロー、Apex、レポート、ページレイアウト）への影響を調査
- 「本当に削除が必要ですか？使わなくなったオブジェクトはタブを非表示にして残す方法もあります」と代替案を提示

### 入力規則（Validation Rules）

#### 作成時

1. `docs/catalog/` で対象オブジェクトの既存入力規則を確認（同じ条件の重複がないか）
2. `docs/requirements/requirements.md` で対応するビジネスルール（BR-XXX）を確認
3. 数式のテストケース（エラーになる値・ならない値）をコメントとして提示

**確認事項:**
- エラーメッセージの表示場所: 項目の上 or ページ上部
- エラーメッセージ文: ユーザーが何をすればいいか分かる文言を提案（「〇〇を入力してください」「〇〇は△△の場合のみ入力できません」等）
- **バイパスパターンが必要か**: 管理者・一括インポート時に回避したい場合は `$Profile.Name` や カスタム権限（`$Permission.Bypass_Validation`）でのバイパス条件を提案
- レコードタイプ限定にするか: 特定レコードタイプのみ適用する場合は `RecordType.DeveloperName` 条件を追加

**変更・削除時:** 既存データへの影響（保存済みレコードが新しいルールに引っかかる可能性）を警告

### Apex

#### 種別選定（「Apex作って」と言われたら最初に用途を確認）

| 用途 | 推奨種別 |
|---|---|
| レコード保存時の処理 | トリガー + ハンドラークラス |
| 5万件超のバッチ処理 | `Database.Batchable` |
| 非同期・軽量処理 / チェーン実行 | `Queueable` |
| 定期実行（夜間バッチ等） | `Schedulable` |
| フローから呼び出す処理 | `@InvocableMethod` |
| 外部システムからのAPI受口 | `@RestResource` |
| 共通ロジックの切り出し | サービスクラス / ユーティリティクラス |

用途が複合している場合（例: 定期実行 + 大量データ）は `Schedulable` が `Database.Batchable` を呼び出す構成を提案する。

#### トリガー設計

- **1オブジェクト1トリガー原則**を必ず守る（複数トリガーの実行順は保証されない）
- トリガー本体はロジックを持たず、全処理をハンドラークラスに委譲する
- before/after・insert/update/delete のどのイベントが必要か確認する
- 再帰防止が必要か確認する（static フラグ or TriggerHandler フレームワーク）

**既存自動化との競合チェック（必須）:**

トリガー作成・変更前に、対象オブジェクトの全自動化を洗い出す:
- 既存Apexトリガー（同一イベント）: `force-app/main/default/triggers/` で対象オブジェクトのトリガーを検索
- レコードトリガーフロー（実行順序: Apex before → Flow before → DB → Apex after → Flow after）: `force-app/main/default/flows/` で検索
- 入力規則（before save で評価）: `force-app/main/default/objects/*/validationRules/` で検索
- ワークフロールール・プロセスビルダー（レガシー。残存していれば警告）

1トリガー/1オブジェクトパターンに従っているか確認。複数トリガーが存在する場合は統合を提案。

#### コーディング規約（常に適用）

- `with sharing` をデフォルト。意図的に無視する場合のみ `without sharing` / `inherited sharing`（理由をコメントで明記）
- SOQL・DML は必ずループ外に配置
- ハードコード禁止（レコードID・メールアドレス・組織固有の値等）→ カスタムメタデータ / カスタム設定を提案
- `Database.insert(records, false)` で部分成功を許容するか確認（デフォルトは全件失敗 `true`）
- FLS確認が必要な場合は `Security.stripInaccessible()` を使用

#### テストクラス（実装とセットで必ず提供）

- `@TestSetup` でテストデータを生成
- `Test.startTest()` / `Test.stopTest()` で対象処理を囲む
- 正常系・異常系・バルク（200件）を必ず網羅
- 外部コールアウトがある場合は `HttpCalloutMock` を実装してセットで提供
- カバレッジ目標: 90%以上（デプロイ要件の75%より高い水準を維持）
- `System.assertEquals` / `System.assertNotEquals` でアサートを必ず入れる（カバレッジだけのテストは禁止）

#### バッチ / Queueable / Scheduled の追加確認

**Batchable:**
- スコープサイズ（デフォルト200）を確認。大きすぎるとCPUタイムアウト、小さすぎると非効率
- 処理間で状態を保持する必要があれば `Database.Stateful` を提案（ヒープ消費に注意）
- エラーレコードの処理方針（スキップして続行 or 全件ロールバック）を確認

**Queueable:**
- チェーン実行の深さ制限（本番5回 / Sandbox無制限）を説明
- `Database.Batchable` との使い分け基準を説明（件数が少ない・チェーンが必要 → Queueable）

**Schedulable:**
- CRON式を提示してユーザーに確認（例: `0 0 2 * * ?` = 毎日午前2時）
- 同時実行が起きる可能性がある場合は排他制御（カスタム設定フラグ）を提案

#### 変更時の影響調査

変更前に以下を調査して結果を提示する:
- 他Apexクラスからのメソッド呼び出し（Grep: クラス名 / メソッド名）
- フローの `Apex Action` での参照
- `@InvocableMethod` / `@AuraEnabled` / `@RestResource` として外部公開されているか（変更の影響範囲が広い）
- APIバージョン（古い場合は最新への更新を提案、ただし強制しない）
- 対応するテストクラスの更新が必要かどうかを確認

#### エラーハンドリング

- `try-catch` は処理単位で適切に設定。例外を握りつぶして空の catch は禁止
- 同じ例外処理が複数箇所に出たらカスタム例外クラスの作成を提案
- エラー発生時の通知・記録方針（Platform Event / カスタムオブジェクトへのログ記録）をユーザーに確認

### Flow

#### 種別選定（「フロー作って」と言われたら最初に用途を確認）

| 用途 | 推奨種別 |
|---|---|
| レコード保存時の項目自動設定・軽量チェック（DMLなし） | レコードトリガーフロー（保存前） |
| レコード保存時の関連レコード更新・メール送信 | レコードトリガーフロー（保存後） |
| ユーザーが手動操作する多段階処理 | 画面フロー |
| 定期実行（毎日・毎月等） | スケジュールフロー |
| 他のフロー・Apexから呼び出す共通処理 | 自動起動フロー（サブフロー用途） |
| Platform Event 受信時 | プラットフォームイベントトリガーフロー |

保存前 vs 保存後の判断基準を説明する:
- **保存前**: 項目値の自動設定・入力チェック。DML不可だが高速
- **保存後**: 関連レコードの作成・更新・外部コール・メール送信が必要な場合

#### 既存自動化との競合チェック（作成前に必ず実施）

`docs/catalog/` と `force-app/` の両方で対象オブジェクトの自動化を確認し、衝突リスクを提示する:

1. `force-app/main/default/triggers/` で対象オブジェクトのApexトリガーを検索
2. `force-app/main/default/flows/` で対象オブジェクトのフローを検索
3. `docs/design/` で設計書に記載された自動化を確認

以下を報告する:
| 種別 | 名前 | 動作タイミング | 競合リスク |
|---|---|---|---|
| Apexトリガー | | before/after insert/update/delete | |
| レコードトリガーフロー | | before/after save | |
| 入力規則 | | save時 | |
| ワークフロールール / プロセスビルダー | | （レガシー） | |

**競合リスクが「高」の場合:**
- 実行順序による影響（Apex before → Flow before → DB → Apex after → Flow after）を説明
- 同一項目を更新する場合は「最後に実行される方が勝つ」旨を警告
- 対処案（条件分岐の追加・統合等）を提示

#### 設計規約（常に適用）

- **ループ内DML禁止**: ループ内ではコレクション変数に蓄積し、ループ外の「レコードを更新」要素で一括処理
- **フォールトパス必須**: DML操作・外部コール・メール送信の全要素にフォールトパスを接続。エラー時の処理（カスタムオブジェクトへのログ記録 or メール通知）をユーザーに確認
- **説明文必須**: フロー全体の目的と、複雑な判定条件・分岐には要素ごとの説明を記載
- **ハードコード禁止**: レコードID・メールアドレス・固定値はカスタムメタデータ / カスタム設定に外出し
- **サブフロー活用**: 他フローと重複するロジックはサブフロー化を提案（再利用・保守性向上）

#### 再帰・無限ループの防止

レコードトリガーフローで自オブジェクトを更新する場合:
- 再帰ループのリスクを警告
- 開始条件（`ISCHANGED` で特定項目の変更時のみ起動）を使って不要な起動を抑制することを提案
- 「すべてのレコード」での起動は避け、条件を絞ることを推奨

#### バージョン管理の注意

- フロー有効化時は新バージョンが作成され旧バージョンが非アクティブ化される
- 大きな変更はサンドボックスで検証してから本番有効化を推奨
- 旧バージョンは削除不可のためバージョンが増えすぎると管理が煩雑になる旨を説明

#### 変更時の影響調査

変更前に以下を調査して提示する:
- 他フローからのサブフロー参照
- Apexからの `Flow.Interview` 呼び出し
- ページレイアウトのアクションボタンに配置されている画面フロー（UI変更の影響）
- スケジュールフローは実行中インタビューが残っている場合、変更が即時反映されない旨を説明

### LWC（Lightning Web Components）

#### 作成時の確認

1. **配置場所**: レコードページ / アプリページ / ホームページ / フロー画面 / Experience Cloud / 他コンポーネントから呼び出し。配置場所によって `targets` と `targetConfigs` が変わる
2. 既存の類似コンポーネントが `force-app/` にないか確認してから作成

#### データアクセスの選定

| 用途 | 推奨手段 |
|---|---|
| レコードの参照・更新 | `@wire(getRecord)` / Lightning Data Service（キャッシュ効く） |
| 関連レコード一覧・複雑な条件 | `@wire` でApexメソッドを呼ぶ |
| ユーザー操作起点の処理（ボタン押下等） | `@AuraEnabled` Apexを imperative call |

`@wire` を優先する理由（キャッシュ・自動再取得・ガバナ制限の節約）を説明した上で用途に応じて選定する。

#### コーディング規約

- **ローディング・エラー状態を必ずハンドリング**: `isLoading` フラグ・`error` 変数をテンプレートに反映
- **SLDSを使用**: `slds-` クラスを使い、独自CSSはレイアウト調整の最小限に留める
- **イベント設計**: 親子間は `CustomEvent`。兄弟コンポーネント間は LMS（Lightning Message Service）または共通の親経由
- `connectedCallback` でリスナー登録した場合は `disconnectedCallback` で必ず解除
- `@api` は外部公開プロパティのみ。内部状態は `@track` 不要（プリミティブ以外はデフォルトでリアクティブ）

#### Apex連携時

- `@AuraEnabled(cacheable=true)` はデータ取得専用。DMLを含む処理には付けない
- エラーは `AuraHandledException` を使って意味のあるメッセージをフロントに返す
- Apexの戻り値の型をJS側で扱いやすい形に設計する（ラッパークラスの活用を提案）

#### 変更時の影響調査

- 同じコンポーネントを使っている他のページ・コンポーネントを調査（Glob: `*.html` 内のコンポーネントタグ参照）
- `@api` プロパティの変更・削除は呼び出し元全てに影響することを警告
- Lightning App Builder に配置済みの場合、デプロイ後に実機での動作確認が必要

#### テスト（Jest）

- ユニットテストは `__tests__` フォルダに配置
- `@wire` のモックは `@salesforce/wire-service-jest-util` を使用
- プロジェクトでJestが導入済みかを確認してから提供する（未導入なら `@salesforce/sfdx-lwc-jest` の設定手順も案内）

## 作業アプローチ — docs を活用した開発

### 着手前のコンテキスト収集（必須）

どんな開発指示を受けても、まず以下を確認する:

```
指示を受ける
  ↓
1. docs/overview/org-profile.md → 用語集で社内用語とSF表現の対応を確認
2. docs/catalog/{対象オブジェクト}.md → 既存の項目・リレーション・入力規則を把握
3. docs/design/{種別}/ → 該当機能の設計書があれば設計に従う
4. docs/requirements/requirements.md → 関連ビジネスルール（BR-XXX）を確認
5. docs/data/master-data.md → ピックリスト値・マスタデータの正確な値を確認
  ↓
着手
```

docs が存在しない場合は確認をスキップして作業する。ただし:
- 「用語集がないため、一般的なSalesforce命名に従います」と伝える
- 作業後に「`/sf-catalog <オブジェクト名>` で定義書を更新することを推奨します」と提案

### 作業後の提案

開発によって組織構成が変わった場合、以下を提案する（自動実行はしない）:
- 項目追加/変更 → 「`/sf-catalog <オブジェクト名>` で定義書の更新を推奨」
- 新機能実装 → 「`/sf-design` で設計書の作成を推奨」
- ビジネスルール変更 → 「`/sf-analyze` で要件定義書の更新を推奨」

### 作業ステップ

1. docs/ から関連コンテキストを収集する
2. 不明点があればリストアップしてユーザーに確認する
3. 影響するオブジェクト・クラス・フローを事前に調査する
4. ガバナ制限リスクとセキュリティリスクを実装前に明示する
5. 実装コードとテストクラスをセットで提供する
6. **`.claude/CLAUDE.md` の Quality Gate に従い、実装完了後に自動でレビューを実行する**（Apex/LWC/Flow/テストクラスは必須）
7. 設定変更が必要な場合は手順を明示する
8. デプロイ前チェックリストを提示する
9. 本番デプロイは必ずユーザー確認を取ってから実行する
